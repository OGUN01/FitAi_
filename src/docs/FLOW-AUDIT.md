## 1. Workout Completeness

### Reality
On workout end, `WorkoutSessionScreen.completeWorkout` (`src/screens/workout/WorkoutSessionScreen.tsx:391`) calls `completionTrackingService.completeWorkout` (`src/services/completionTracking.ts:140`). Calories are computed via the MET calculator (`calculateWorkoutCalories` in `src/services/calorieCalculator.ts:171`) using the user's resolved weight — no hardcoded fallback (returns 0 if weight missing, per CLAUDE.md #8). The Supabase `workout_sessions` row is inserted with `calories_burned`, `total_duration_minutes`, `exercises_completed` (logged sets), `rating`, `notes`, `is_completed=true`, and `workout_plan_id`. Per-set data (weight/reps/rpe) is dual-written to `exercise_sets` via `_writeExerciseSets`. Partial exits are handled by `savePartialExit` (`completionTracking.ts:991`) which persists logged sets and marks `is_completed=false`. The store is updated via `addCompletedSession` (idempotent by `sessionId`), streak is updated via `achievementStore.updateCurrentStreak`, and achievements are tracked. A reactive deload check runs post-completion using `exerciseHistoryService` history.

### Bugs
- **`src/services/completionTracking.ts:105-107`** — `calculateActualCalories` returns `0` when weight is unavailable but does NOT `console.warn` (unlike `calculateWorkoutCalories` at `calorieCalculator.ts:148` which does warn). The completion path swallows the missing-weight case silently at the `calculateActualCalories` layer. **Severity: Medium** (violates CLAUDE.md #5 — the inner MET function warns, but the outer wrapper that the completion flow actually calls does not, so a developer tracing a zero-calorie workout sees no log at the completion layer).
- **`src/services/completionTracking.ts:84-89`** — `calculateActualCalories` short-circuits to `sessionData.stats.caloriesBurned` if the screen pre-computed it, bypassing the MET calc entirely. This is intentional (screen-computed values are authoritative) but creates a SSOT ambiguity: the `WorkoutSessionScreen` computes `finalStats.caloriesBurned` from `session.workoutStats` which is derived in the session hook, NOT from `calorieCalculator`. If the session hook's calorie calc diverges from `calculateWorkoutCalories`, the persisted value won't match what the MET calculator would produce. **Severity: Low** (currently the session hook uses the same MET calc per P1-8 fix at `completionTracking.ts:725`, but the coupling is implicit, not enforced).
- **`src/services/completionTracking.ts:289-296`** — `_writeExerciseSets` is called fire-and-forget (`.catch(...)` only logs). If it fails, the `exercise_sets` rows are lost with no offline queue/retry (unlike `workout_sessions` which has BUG-4 queue fallback). A network blip during set-write means the user's logged weight/reps/rpe never reach Supabase. **Severity: Medium** (partial data loss on flaky network; the store SSOT retains it locally, but cross-device sync loses it).
- **`src/services/completionTracking.ts:751-753`** — `updateWorkoutProgress` auto-completes at `progress >= 100` by calling `completeWorkout(workoutId, exerciseData, userId)`. The `exerciseData` is passed as `sessionData`, which means `sessionData.sessionId` is undefined unless the caller explicitly sets it. If a caller increments progress to 100 without a `sessionId`, `completeWorkout` inserts a fresh `workout_sessions` row with a generated UUID `sessionId` (`completionTracking.ts:362`) — correct, but the auto-complete path does NOT carry the logged exercise sets from `currentWorkoutSession` (it passes `exerciseData` as the opaque `sessionData`, which lacks `stats.exercises`). **Severity: Medium** (the `exercises_completed` field will be empty for auto-completed workouts via the progress path, losing set-level data).
- **`src/services/completionTracking.ts:300-303`** — `analyticsDataService.updateTodaysMetrics` accumulates `caloriesBurned` into `analytics_metrics`. The WorkoutSessionScreen comment (`WorkoutSessionScreen.tsx:460-465`) explicitly calls out the P0 double-count fix removing `analyticsHelpers.trackWorkoutCompleted`. However, if `fitnessRefreshService.refreshAfterWorkoutCompleted` (`completionTracking.ts:314`) also triggers a data reload that re-fires a realtime event, `loadData()` could re-process the session. The `addCompletedSession` idempotency guard (`fitnessStore.ts:559-565`) prevents double-counting in the store, but `analytics_metrics` accumulation (`updateTodaysMetrics` with `workoutsCompleted: 1`) is NOT idempotent — a re-fire would double-count the analytics row. **Severity: Medium** (relies on the realtime handler not re-firing; fragile).

### Improvements
- **[min]** Add `console.warn` in `calculateActualCalories` (`completionTracking.ts:105`) when weight is unavailable, mirroring `calculateWorkoutCalories`.
- **[min]** Queue `_writeExerciseSets` via `offlineService` on failure (same pattern as the `workout_sessions` BUG-4 fix at `completionTracking.ts:268-281`).
- **[med]** Make `analyticsDataService.updateTodaysMetrics` idempotent for `workoutsCompleted`/`caloriesBurned` by keying on `session_id` (add a `session_id` column to `analytics_metrics` or use a dedup set in the context JSONB).
- **[med]** Fix the `updateWorkoutProgress` auto-complete path (`completionTracking.ts:751-753`) to pull `currentWorkoutSession.exercises` from the store and pass it as `sessionData.stats.exercises` so set-level data is persisted.
- **[max]** Unify calorie computation: remove the `sessionData.stats.caloriesBurned` short-circuit in `calculateActualCalories` and always recompute via `calculateWorkoutCalories` using the logged `exerciseProgress` (which `calculateCompletedWorkoutCalories` at `calorieCalculator.ts:244` already supports). This eliminates the implicit coupling between the session hook's calorie calc and the completion service's calc.

## 2. Progressive Overloading

### Reality
"Progressive overload" is largely a **schema field and text note, not an enforced mechanism**. The `prefersVariety`, `stressLevel`, and `activityLevel` fields (`fitai-workers/src/utils/validation.ts:170-173`) ARE consumed — but only for **split selection scoring** (`fitai-workers/src/utils/workoutSplits.ts:665-729`): `stressLevel` adjusts recovery-demand scoring, `activityLevel` adjusts recovery capacity, `prefersVariety` biases toward higher-day splits. The `assignWorkoutParameters` function (`fitai-workers/src/utils/workoutStructure.ts:233`) assigns sets/reps/rest based on **experience level + fitness goal + medical modifications ONLY** — it does NOT accept `weekNumber` and does NOT scale volume or intensity week-over-week. The `weekNumber` (mesocycle week 1-4, from `useFitnessStore.getMesocycleWeek()`) is used in exactly two places: (1) the cache key (`workoutGeneration.ts:415`) so different weeks get different cached plans, and (2) `generateProgressionNotes` (`workoutStructure.ts:429-448`) which emits **text-only guidance** ("Week 2: Increase weight by 5-10%...") — no actual parameter change. The AI prompt path (`buildWorkoutPrompt` at `workoutGeneration.ts:98-365`) never receives `weekNumber` and contains only generic "progression" text. Critically, **no previous workout history is fed into generation**: `transformForWorkoutRequest` (`src/services/aiRequestTransformers.ts`) does not send `completedSessions`, `workout_sessions`, or any prior performance data. Each generation is blind to history — there is no closed loop.

### Bugs
- **`fitai-workers/src/utils/workoutStructure.ts:233-296`** — `assignWorkoutParameters` ignores `weekNumber` entirely. Week 1 and Week 4 produce identical sets/reps/rest for the same exercise classification + experience + goal. The "progressive overload" documented in `FITAI_DATA_ARCHITECTURE.md` §F.3 (Double Progression, RPE modulation, deload) is **not implemented in the generation engine** — it exists only as architecture-doc prose. **Severity: High** (the product claims progressive overload but delivers static plans).
- **`fitai-workers/src/utils/workoutStructure.ts:429-448`** — `generateProgressionNotes` emits week-specific text ("Week 3: Push intensity - aim for RPE 7-8") but the generated workout's actual sets/reps/weight do not reflect this. The note is aspirational, not actionable. **Severity: Medium** (misleading to the user; the note implies the workout is calibrated for that week's intensity).
- **`src/services/aiRequestTransformers.ts`** — No `completedSessions`, `workout_sessions`, PRs, or prior set data is sent to the Worker. The AI prompt (`workoutGeneration.ts:260-364`) has no knowledge of what the user lifted last week, whether they hit the top of the rep range, or whether they failed sets. **Severity: High** (closed-loop progressive overload is impossible without this; each plan is generated in a vacuum).
- **`fitai-workers/src/handlers/workoutGeneration.ts:403-405`** — `regenerationSeed` defaults to a random value when absent, so every "regenerate" busts the cache. But `weekNumber` is included in the cache key (`workoutGeneration.ts:415`) — if a user regenerates in the same mesocycle week, they get a different plan, but it won't be "harder" than the previous one. There's no monotonic progression guarantee. **Severity: Low** (cache behavior is correct; the issue is the absence of progression logic, not the cache).
- **`src/hooks/useFitnessLogic.ts:428`** — `mesocycleWeek` is clamped to `[1, 4]` and sent to the Worker, but the Worker only uses it for the cache key + progression notes text. After week 4, the mesocycle resets (per `checkProactiveDeload`), but there's no mechanism to carry forward the user's last-known weights into the next mesocycle's week 1. **Severity: Medium** (each new mesocycle starts from scratch, ignoring the user's accumulated strength).

### Improvements
- **[min]** Pass `weekNumber` into `assignWorkoutParameters` and scale volume/intensity: e.g., week 2 = +1 set on compounds or +5% rep-range ceiling, week 3 = RPE target in notes becomes a rep-range adjustment, week 4 = -30% volume (deload). This makes the progression notes truthful.
- **[min]** Add `weekNumber` to the AI prompt (`buildWorkoutPrompt`) with explicit instructions: "This is week N of a 4-week mesocycle. Week 2: increase weight 5-10% from week 1. Week 3: peak intensity. Week 4: deload (reduce volume 30%, weight 20%)."
- **[med]** Send the user's last completed workout for each exercise (from `exercise_sets` / `exercise_prs`) to the Worker, and inject it into the prompt: "Last session: Bench Press 60kg×8,8,8. This week: target 62.5kg or 60kg×10." This creates the closed loop.
- **[med]** Implement the Double Progression system documented in `FITAI_DATA_ARCHITECTURE.md` §F.3: when ALL sets hit the top of the rep range, auto-suggest a weight increase in the next plan; RPE 1 (Easy) = double jump, RPE 3 (Hard) = hold; two consecutive failed sessions → deload to 90%. The data is already in `exercise_sets` (weight_kg, reps, rpe) — the logic just isn't wired.
- **[max]** Persist the user's working weights per exercise (from `exercise_prs` or a new `exercise_working_weights` table) and feed them into both the rule-based and AI generation paths. This makes progression deterministic rather than LLM-guessed.

## 3. Diet Saving

### Reality
After AI generates a weekly meal plan, `useMealPlanning.handleMealPlanResult` (`src/hooks/useMealPlanning.ts:180`) calls `nutritionStore.saveWeeklyMealPlan` (`src/stores/nutritionStore.ts:302`). The store writes to Zustand first (`set({ weeklyMealPlan: plan })` at line 308), then validates non-empty meals (line 311-313 returns early with a `console.warn` if empty — no DB write, no user alert), then queues a Supabase upsert to `weekly_meal_plans` via `offlineService.queueAction` (line 421-429). The DB schema (`supabase-types.generated.ts:2834`) matches the write payload (`plan_data`, `plan_title`, `plan_description`, `total_calories`, `total_meals`, `week_number`, `is_active`). On reload, `loadWeeklyMealPlan` (`nutritionStore.ts:445`) fetches the active plan from Supabase by `is_active=true`, extracts `plan_data` JSONB, and sets it in the store with `databaseId`. A single-active-plan invariant is enforced by a partial unique index (`20260319000004_enforce_single_active_weekly_meal_plan.sql:28-30`). Users CAN log actual meals eaten vs planned: `completeMeal` (`completionTracking.ts:457`) inserts a `meal_logs` row with `from_plan=true`, `plan_meal_id`, `is_completed=true` (P2-11 fix), and the plan's food items. The `meal_logs` table is the SSOT for consumed nutrition; `weekly_meal_plans` is the SSOT for planned meals.

### Bugs
- **`src/stores/nutritionStore.ts:311-313`** — `saveWeeklyMealPlan` returns early with only `console.warn` when `plan.meals` is empty. The caller (`useMealPlanning.handleMealPlanResult` at `useMealPlanning.ts:185`) does NOT check the return value and proceeds to `setWeeklyMealPlan(weeklyPlan)` and show the "Meal Plan Generated!" alert. **Severity: High** (the user sees a success alert but the plan was never saved to the DB; on reload it vanishes. This is the "empty-plan silent return" bug).
- **`src/hooks/useMealPlanning.ts:187-189`** — `handleMealPlanResult` catches `saveErr`, logs it, and `throw`s. But `saveWeeklyMealPlan` catches its own DB errors internally (`nutritionStore.ts:430-442`) and does NOT re-throw (it sets `planError` state and returns normally). So the `throw saveErr` in `handleMealPlanResult` will NEVER fire for DB-save failures — only for the local-store `set()` failure (which is extremely rare). **Severity: High** (DB save failures are swallowed in the store; the user sees "Meal Plan Generated!" even when Supabase save failed. The `planError` state is set but no UI surfaces it — `useMealPlanning` never reads `planError`).
- **`src/stores/nutritionStore.ts:308` vs `:399`** — `saveWeeklyMealPlan` calls `set({ weeklyMealPlan: plan })` at line 308 (before DB validation), then later calls `set({ weeklyMealPlan: planDataWithDbId })` at line 399 (after DB lookup). The second write is correct (it attaches `databaseId`), but the first write uses the plan WITHOUT `databaseId`. If the DB write fails, the store retains the plan without a `databaseId`, so a subsequent `saveWeeklyMealPlan` call (e.g., from `persistData` at line 846) will do a CREATE instead of an UPDATE, creating a duplicate row (mitigated by the single-active-plan index, but the old row is deactivated rather than updated). **Severity: Medium** (correctness is preserved by the index, but the store/DB `databaseId` mismatch is a latent inconsistency).
- **`src/stores/nutritionStore.ts:328-442`** — The entire Supabase save is wrapped in `offlineService.queueAction`, which is fire-and-forget. If the queue action fails after 3 retries (`maxRetries: 3`), the plan is permanently lost from Supabase. There is no user-facing error and no retry-forever path. **Severity: Medium** (silent data loss after retry exhaustion; the user's plan exists only in AsyncStorage).
- **`src/stores/nutritionStore.ts:472-480`** — `loadWeeklyMealPlan` clears the local plan (`set({ weeklyMealPlan: null })`) when the DB returns no active rows. If the DB query fails transiently (network timeout), the `error` branch at line 483 catches it and falls through to the meal-logs fallback (line 491) — but if the query returns `error: null` with an empty array (e.g., RLS policy issue or race condition during signup), the local plan is wiped. **Severity: Medium** (a transient empty-result from Supabase wipes the user's locally-generated plan before it's persisted).
- **`src/stores/nutritionStore.ts:50-52` in `createLoggedFoodsFromMealItems`** (`completionTracking.ts:34-52`) — When a planned meal is completed, the food items are copied from the plan's `meal.items` with `calories: item.calories ?? 0`. If the AI returned meals with missing `calories` on individual items (the `totalCalories` is on the meal, not always per-item), the logged food entries will have `calories: 0`, making per-item calorie tracking inaccurate. **Severity: Low** (the meal-level `total_calories` is correct in `meal_logs`; only the per-food-item breakdown is affected).

### Improvements
- **[min]** Make `saveWeeklyMealPlan` return a `boolean` (or throw on DB failure) and have `handleMealPlanResult` check it before showing the success alert. If save failed, show an error alert instead.
- **[min]** Surface `planError` in the DietScreen UI (a banner or toast) so users know when a save failed. Currently `planError` is set but never read by any component.
- **[min]** Fix the empty-plan early return (`nutritionStore.ts:311-313`): throw an error instead of silently returning, so `handleMealPlanResult`'s catch block fires and the user sees "Generation produced an empty plan."
- **[med]** Don't `set({ weeklyMealPlan: null })` in `loadWeeklyMealPlan` on empty DB result — keep the local plan and log a warning. The DB is the SSOT for *persisted* plans, but a transient empty result shouldn't wipe an in-flight local plan.
- **[med]** Add a `maxRetries: Infinity` (or a longer backoff) for `weekly_meal_plans` queue actions, since losing a user's entire weekly meal plan is high-cost. Alternatively, add a "plan sync pending" indicator that persists until the DB write succeeds.
- **[max]** Unify the dual-write: `saveWeeklyMealPlan` writes to Zustand (line 308) AND queues a DB write (line 421), but these are not atomic. If the app crashes between the two, the store has the plan but the DB doesn't. Consider writing to DB first (await), then updating the store on success (with `databaseId`), reverting the store on failure. This matches the "DATABASE-FIRST PATTERN" already used in `completeMeal` (`nutritionStore.ts:548`).

## Top 5 Cross-Cutting Fixes (Priority Order)

1. **[P0] Surface diet-save failures to the user.** `saveWeeklyMealPlan` swallows DB errors (`nutritionStore.ts:430-442`); `handleMealPlanResult` shows a success alert regardless (`useMealPlanning.ts:194-199`). Users lose plans silently on DB failure. Fix: return failure from the store, check it in the hook, show an error alert. Also fix the empty-plan early return (line 311) to throw instead of silently returning.

2. **[P0] Implement real progressive overload in `assignWorkoutParameters`.** Currently `weekNumber` only affects cache key + text notes (`workoutStructure.ts:429-448`). Volume/intensity is static across all 4 mesocycle weeks. Pass `weekNumber` into `assignWorkoutParameters` and scale sets/reps/rest per week (week 2 = +volume, week 3 = +intensity, week 4 = deload). This makes the documented Double Progression system (`FITAI_DATA_ARCHITECTURE.md` §F.3) real instead of aspirational.

3. **[P1] Feed prior workout history into generation (closed loop).** `transformForWorkoutRequest` sends zero workout history. The AI prompt and rule-based engine generate each plan blind to what the user previously lifted. Fix: send last completed session per exercise (from `exercise_sets`) + current PRs (from `exercise_prs`) to the Worker, inject into the prompt ("Last bench: 60kg×8. Target: 62.5kg or 60kg×10"), and use in `assignWorkoutParameters` to set starting weights. Without this, progressive overload is impossible even after fix #2.

4. **[P1] Make `_writeExerciseSets` resilient with offline queue.** `completionTracking.ts:289-296` is fire-and-forget with only `console.error` on failure. Set-level data (weight/reps/rpe) is lost on network failure with no retry. Fix: queue via `offlineService` (same pattern as the `workout_sessions` BUG-4 fix at line 268-281). This is the data progressive overload depends on — losing it breaks the closed loop in fix #3.

5. **[P2] Make `analyticsDataService.updateTodaysMetrics` idempotent for workout completions.** `completionTracking.ts:300-303` accumulates `caloriesBurned` non-idempotently. If `fitnessRefreshService.refreshAfterWorkoutCompleted` (line 314) triggers a reload that re-fires a realtime event, the analytics row double-counts. The store-level `addCompletedSession` is idempotent (`fitnessStore.ts:559`), but the analytics accumulation is not. Fix: key the analytics update on `session_id` (add column or use context JSONB dedup set).
